# 结构化表达图形模板库

这些模板是可复用的图形模式。每个模板包含适用场景、内容拆解方式和推荐输出格式。

## A1 用户主路径流程图

用于 PRD、业务文档或流程说明中解释用户 / 角色如何完成一个核心目标。

输入信号：
- 内容里有用户目标、入口、动作序列、决策分支、成功状态或失败状态。
- 用户提到流程图、用户路径、产品体验流程。

结构：
1. 触发条件 / 入口。
2. 主路径动作。
3. 决策点或校验点。
4. 成功结果。
5. 异常分支。

输出：
- 草稿优先用 Mermaid flowchart。
- 文档中需要二次编辑时用 draw.io XML。
- 需要贴到文档时补 SVG 预览。

设计规则：
- 主路径要视觉上最明显。
- 异常路径要更细或更浅。
- 节点标签用“动作 + 状态”，不要写长段描述。

## A2 角色泳道流程图

用于跨角色流程、运营流程，或涉及用户、系统、运营、客服、销售、财务、第三方服务的业务 / 产品文档。

输入信号：
- 多个角色分别负责不同步骤。
- 交接、审批、升级、人工介入很重要。
- 用户需要对齐研发、运营、业务、销售、客服、财务或管理团队。

结构：
1. 按责任方定义 3-6 条泳道。
2. 每个动作放在对应负责人的泳道里。
3. 标出交接点、阻塞点。
4. 突出关键路径。
5. 只在必要时加入异常或回退路径。

输出：
- 最终可编辑版本优先用 draw.io XML。
- 快速草稿可用 Mermaid flowchart + subgraph。
- 业务汇报可转成 PPT。

设计规则：
- 泳道代表责任边界。
- 用一个强调色标记交接点或瓶颈。
- 不要把技术实现细节塞进业务泳道。

## A3 页面 / 状态流转图

用于页面、订单、审批、任务、工单、生命周期状态，或权限 / 状态变化。

输入信号：
- 内容里有草稿、待审核、已通过、已驳回、生效、过期、取消等状态。
- 用户问“有哪些状态”或“状态如何变化”。

结构：
1. 列出所有状态。
2. 找出状态之间的触发动作。
3. 标记终态。
4. 标记回退、重试或异常流转。
5. 只有权限影响状态变化时才加入权限。

输出：
- Mermaid state diagram。
- 规格说明或流程说明用 draw.io XML。

设计规则：
- 节点是状态，边是动作。
- 回退或异常流转用强调色。

## B1 指标拆解树

用于数据分析、KPI 设计、增长诊断，或解释某个指标为什么变化。

输入信号：
- 有北极星指标、目标指标或 KPI。
- 用户提到驱动因素、原因、指标拆解、增长、下降、机会测算。

结构：
1. 北极星指标或目标指标。
2. 3-5 个一级驱动因子。
3. 二级可度量驱动因子。
4. 行动项和负责人 / 团队映射。
5. 如果用户已有证据，标记最可疑的驱动因子。

输出：
- 快速拆解用 Mermaid mindmap 或 flowchart。
- 战略 / 报告页面用 PPT 或 SVG。

设计规则：
- 每个叶子节点都要能对应动作或可度量信号。
- 除非输出知识地图，否则不要超过 3 层。

## B2 漏斗诊断图

用于转化、 onboarding、激活、下单、付费、留资等掉点问题。

输入信号：
- 内容包含步骤和转化率。
- 用户想知道用户在哪里流失，或如何提升转化。

结构：
1. 按顺序列出漏斗步骤。
2. 每一步的人数或转化率。
3. 步骤之间的掉点。
4. 如果有数据，加入分群对比。
5. 针对最大掉点给出假设和行动建议。

输出：
- 先给 Markdown 表格 + 图表结构。
- 最终汇报用 SVG 或 PPT。

设计规则：
- 不要只画漏斗，必须附诊断结论。
- 最大掉点要高亮。

## B3 留存 / 分群热力图

用于留存、复访、渠道质量、版本影响、用户分群对比。

输入信号：
- 用户提到留存、cohort、渠道、版本、用户群、月批次、重复行为。

结构：
1. 行：cohort 或用户分群。
2. 列：周期或生命周期天数。
3. 单元格：留存、转化、收入或活跃。
4. 侧边注释：关键模式和可疑原因。

输出：
- 数据检查先用 Markdown 表格。
- 汇报用 SVG / PPT 热力图。

设计规则：
- 使用单一渐进色阶。
- 必须附一句人话结论。

## C1 竞品能力矩阵

用于竞品分析、功能对比、供应商选择、能力差距分析。

输入信号：
- 内容列出了竞品和对比维度。
- 用户比较功能、能力、价格、集成、UX、生态、AI 能力或 GTM。

结构：
1. 竞品作为列。
2. 能力维度作为行。
3. 如果用于决策，加入权重或重要度。
4. 加证据备注或置信度。
5. 输出机会点总结。

输出：
- 先给 Markdown 表格。
- 最终汇报用 PPT 表格或 SVG。

设计规则：
- 优先用评分 / 权重，不要只打勾。
- 行维度按 3-5 个能力类别分组。

## C2 定位四象限

用于战略定位、市场空位判断、竞品分群、方案叙事。

输入信号：
- 用户想表达相对定位。
- 内容里存在两个有意义的决策变量。

结构：
1. 根据真实决策逻辑选择 x 轴和 y 轴。
2. 放置竞品、我方方案或其他对象。
3. 给四个象限命名。
4. 标出市场空位或目标迁移方向。
5. 解释为什么选择这两个轴。

输出：
- 优先 PPT / SVG。
- 绘图前先给 Markdown 坐标表。

设计规则：
- 坐标轴必须是决策变量，不能只是“高 / 低”装饰。
- 高亮推荐战略区域。

## C3 市场 / 生态地图

用于市场格局、生态策略、伙伴地图、渠道策略、替代方案分析。

输入信号：
- 内容包含直接竞品、间接竞品、替代方案、伙伴、上下游、渠道或用户替代选择。

结构：
1. 中心：用户问题、我方方案或核心对象。
2. 内圈：直接竞品或核心替代方案。
3. 外圈：替代方案、渠道、伙伴、基础设施。
4. 用标签说明关系类型。
5. 高亮战略威胁或机会。

输出：
- 战略页优先 SVG / PPT。
- 需要可编辑生态图时用 draw.io。

设计规则：
- 按关系类型组织，不要随机摆品牌。

## D1 战略路线图

用于战略规划、季度规划、转型路径、版本规划、能力建设节奏。

输入信号：
- 内容有阶段、时间线、里程碑、优先级或先后顺序。

结构：
1. 按时间或成熟度划分阶段。
2. 每个阶段的目标。
3. 关键举措。
4. 里程碑和依赖关系。
5. 如果有数据，加入指标或负责人。

输出：
- 高管汇报优先 PPT 结构。
- 可补 SVG 预览。
- 简单路线图草稿才用 Mermaid。

设计规则：
- 路线图不是任务清单，每个阶段要以结果为中心。

## D2 能力模型 / 冰山分层

用于 AI 能力模型、平台能力层级、组织能力建设，或解释显性价值背后的系统支撑。

输入信号：
- 内容区分显性体验、系统支撑、数据、工具、流程、人员、治理或基础设施。
- 用户想要“冰山模型”“分层模型”或“能力模型”。

结构：
1. 显性层：用户可见体验和结果。
2. 系统层：方案、流程、自动化、运营机制。
3. 基础层：数据、基础设施、治理、人才、工具链。
4. 必要时加入成熟度阶段或案例。

输出：
- 优先 SVG / PPT。
- Mermaid 通常只适合做粗略大纲。

设计规则：
- 上层解释用户看见什么。
- 下层解释它为什么能成立。
- 标签要短，细节放侧边注释。

## D3 全景信息地图

用于高密度知识地图、学习路线、业务架构图、产品架构图、能力全景，或一页式参考图。

输入信号：
- 内容很广、层级多，目标是沉淀和学习，而不是短会快速汇报。
- 用户喜欢“一张大图讲全局”。

结构：
1. 左侧或顶部放标题锚点。
2. 用 3-6 个横向分区或主域。
3. 每个分区内放模块组。
4. 可增加侧边区域：推荐、资源、案例、行动。
5. 分组标签和间距要统一。

输出：
- 优先 SVG / draw.io，方便沉淀和二次编辑。
- 如果用于 PPT，应拆成多页。

设计规则：
- 全景图可以高密度，但一级分组必须清楚。
- 如果用于高管汇报，拆成 3-5 页。
